4. Twilio SDK 操作時序

Twilio SDK 屬於 client-side library，沒有對外 HTTP 呼叫（純 WebSocket + per-instance method）。因 page.evaluateOnNewDocument 的 fetch/WS hook 沒能在 trace reload 後保留，本節以 source-level 分析 為主，搭配實際觀察到的 network/timing 佐證。

4.1 初始化流程（推導自 TwilioClientContext/provider.jsx）

  1. 取得 Twilio JWT：POST /api/auth/twilio/token（×2）
  2. Client.create(token)：載入 @twilio_conversations.js，建立 WS（觀察到 3 個 WS connection）
  3. getSubscribedConversations() 分頁迴圈 — provider.jsx:352-364
    page.items → push → await page.nextPage() → loop
    每頁通常 50 個 conversation，回完才釋出 → 串行
  4. getConversationMeta(sids) 批次呼叫後端 — provider.jsx:240 → 觀察到 7 次 batch call（見 §3.2）
  5. Per-conversation 多 promise — provider.jsx:245-310，每個 conversation 都跑：
    • conversation.getParticipants()
    • conversation.getUnreadMessagesCount()
    • conversation.getMessages(10) — 拉回最後 10 則訊息
    即使用 Promise.all 包起來，整個 batch 對 350 個 conversation 仍是 1050 個 async op。對 2000 個案：6000 個 async op。

5. Twilio 人數上限（per-user conversations）

根據 Twilio Conversations Classic limits： 單一 user identity 最多 1,000 個 conversation。個管師被自動加進每一個個案的聊天室 → 在 1000 個案就會撞牆，2000 個案必然失敗。

選項做法工程量對 perf 影響
Lazy join（推薦先試）個管師建立個案時不自動 add；打開該聊天室時才 add，關閉後保留S同時解決上限 + 大幅減少 subscribed conversations
共用 role identity使用 category-{id} 共用身份，而非個人 identityM解上限，但失去細粒度發言記錄
Backend proxy個管師完全不 join；訊息由 backend system identity 代發代讀L解上限，但失去 Twilio 原生 push / typing indicator
自動清理定期把長期未互動的 conversation 移除個管師S輔助手段，搭配 lazy join 效果最好

6. Recommendations / Action Plan

1
ChatList 加上 virtualization effort: S ✓ DONE — 見 §8
apps/web/case-management/src/containers/Chat/ChatList/index.jsx:209 — 直接 .map() 渲染所有 conversations

用 react-window / react-virtuoso 取代 .map()。固定 row height 用 FixedSizeList，2 小時內可上線。 對 first-render 時間是線性下降：350 row → 只 render 視窗內 ~15 row，DOM 從 3,340 降至 ~1,500，2000 個案場景也只 render 15 個。

2
Lazy join Twilio conversation effort: M P0
backend/src/chat/routes/conversation.ts: POST /chat/conversations (line 35-81) — 建立個案時自動 add case manager

停用「建立個案時自動 add 個管師」。改為「個管師 open chat room 時才 add」（API: PUT /chat/conversations/:sid/participants）。 立刻解 Twilio 1000-conv 上限，且大幅縮減 getSubscribedConversations() 回傳數量 — 也就是縮減本報告全部下游 N+1 痛點。 需要設計：未加入時，個管師仍能透過 backend admin endpoint 看到列表概覽（不訂閱 WS）。

3
把 ChatList metadata 改成 server-side 分頁 + viewport-driven effort: M P0
provider.jsx:240, conversation.ext.ts:76-114

後端 GET /api/chat/conversations 加上 ?page=&pageSize=。前端搭配 virtualization 只請求視窗內 + buffer 的 conversation metadata（含 last message + unread）。 2000 個案場景只會抓回 ~30 筆 → 從 7 MB 降到 ~5 KB。

4
Per-conversation Twilio op 改成 on-demand effort: M P1
provider.jsx:245-310 — 對每個 conversation 都跑 getUnreadMessagesCount + getMessages(10) + getParticipants

初次載入只取 視窗內那幾個 conversation 的 unread / last message；其餘等使用者 scroll 到再 fetch。 把 6000 個 promise 砍到 ~30 個。Twilio SDK 對「未訂閱的 conversation」也支援 client.peekConversationBySid() 等輕量介面。

5
未讀只顯示紅點，不顯示數字 effort: XS ✓ DONE — 見 §9
provider.jsx:293 — conversation.getUnreadMessagesCount() per conv

把 unread badge 從「5」改成「●」（has-unread / no-unread）。 這是一個 UX 退一步換 perf 大躍進 的取捨：

  • 完全不必呼叫 getUnreadMessagesCount() — 一個沉重的 Twilio SDK 端方法（內部會 fetch read-horizon + 計算未讀區間）。
  • 「有沒有未讀」只需要比對 lastMessageTime vs lastReadTimestamp，兩個欄位都已經在 conversation metadata 或可由後端一次算完。
  • 2000 個案場景：~2000 個 promise 直接歸零，比 #4 on-demand 還省（連視窗內的 ~15 個都不必算）。
  • 與 #5（後端 summary）合併效果最大 — backend 直接回 has_unread: boolean，前端零計算。
UX 風險：使用者看不到精確數字。多數現代 IM（Slack、Teams、LINE 在某些場景）都用紅點處理長尾，可考慮列表用紅點、進到聊天室再顯示確切「↑ N 則未讀」。

6
Unread / last message 改由後端 push effort: M-L P1

後端維護一張 conversation_summary（last_message, last_at, has_unread_per_user）。前端打一支 GET /api/chat/summary?page=... 即取得 ChatList row 需要的全部欄位。完全不必依賴 Twilio SDK 來算 unread / 撈 last message。 效果：單一 HTTP request 取代數千個 WS round-trip。與 #5 合用時，後端只需回 boolean。

7
Defer / dynamic-import 3rd-party SDK effort: XS P2

Google Tag Manager（478 KB）/ LINE LIFF（146 KB）/ Bugsnag — 用 async defer 或 React.lazy 延後到 idle 後載入。不會直接降 perceived latency 太多，但減少主執行緒競爭。

8
定期清理 / archive effort: S P2

對長期無互動（如 6 個月）的 conversation，在 cron job 把個管師從 participants 中移除（但保留訊息）。同時提供「重新加入」的 UX flow。輔助方案，搭配 #2 的 lazy join 形成完整生命週期管理。

為什麼 LCP / FCP / Load 沒改善反而略升？
LCP element 是 nodeId 235 — 頂部 banner 的「個案管理協作平台」MuiTypography-h5 heading link，不在 ChatList 內。 Virtualization 只動 ChatList 的渲染量，動不到 banner 出現時機 — LCP 反映的是「banner heading 何時可見」，主要受 JS bundle parse 順序影響。 多出來的 ~100-500ms 是：
  • local dev mode 量測 noise（HMR cache state 每次不同、TTFB 從 5ms 跳到 26ms）
  • 新增 react-window 與 ResizeObserver setup 的 JS 工作量（~10ms 級別）
  • 同 trace 重跑變動範圍本來就 ±200ms 起跳
真正能驗證 LCP 改善需要：(a) production build（dev mode 470+ ESM module 載入 dominates）、(b) 改 LCP element 不在 ChatList 內的這個事實 — 加上 #2/#3 之後 banner 不需要等 Twilio subscribe，LCP 才會明顯下降。

8.3.1 #1 真正解決的是什麼

#1 的核心價值不在「縮短 first paint」而在「擴大 scalability headroom」：

  • DOM size 線性 → 常數：350 conv 與 2000 conv 的 DOM 都會是 ~12 個 row（只有可視區渲染）
  • JS heap 線性 → 常數：未來 2000 conv 不會出現 600MB+ 高峰 / OOM
  • Scroll 效能：350 row 滑動順、2000 row 也順
  • Re-render cost 線性 → 常數：state 變動只 reconcile 視窗內 row（這個在現在 350 conv 不容易感覺到，2000 conv 會非常明顯）

load 時間沒明顯改善是合理的：trace span 5.66s 主要被 Twilio paginated subscribe + 7 次 /api/chat/conversations batch 吃掉，這兩個是網路層瓶頸，要 #2 (lazy join) / #3 (server-side 分頁) / #5 (紅點化) 才能下降。
