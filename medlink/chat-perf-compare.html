#1+#5 把「能不能撐住 2000 conv」從「很可能崩（OOM 風險、26,000 DOM nodes）」變成「能撐住，但等很久（網路串行 ~1 分鐘）」。 要把 chat list ready 壓到 <5s 需要：#2 lazy join（只 subscribe 視窗內 conv）+ #3 server-side 分頁（只抓視窗內 metadata）。

4. Trade-off & 已知 UX 改變

  • 未讀 badge 變紅點：個管師看不到精確未讀數字（5、12…），只看到「有 / 沒有」。如果想看精確數字仍可進到 chat room 內 message list 直接看。
  • Tab 不再顯示未讀總數：「個案 12 / 醫事人員 4」變成「個案 ● / 醫事人員 ●」。個管同事如果原本依賴這個數字做排序判斷，需要溝通。
  • 大量未讀的 conv 延後抓訊息：超過 10 則的未讀訊息現在「進到 chat room 才會 fetch 進來」（之前是 app 啟動就全抓）。如果使用者習慣打開 app 就能看到所有歷史訊息，現在會在 chat room 首次 open 時稍有 lag。
  • 不可逆性低：所有變動都是純前端、無 DB schema 變動、無 API breakage — revert 任一 commit 即恢復。
